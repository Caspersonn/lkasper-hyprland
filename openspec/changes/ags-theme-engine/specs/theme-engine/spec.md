## ADDED Requirements

### Requirement: Theme engine reads Base16 palette JSON
The theme engine SHALL read palette files from `~/.local/share/lkasper-hyprland/themes/<name>/colors.json`. Each JSON file contains keys `base00` through `base0F` (bare hex without `#`), plus `background`, `foreground`, and `accent` (with `#` prefix). The engine SHALL enumerate available themes by listing directories under `~/.local/share/lkasper-hyprland/themes/`.

#### Scenario: Load a valid theme palette
- **WHEN** `loadTheme("gruvbox")` is called
- **THEN** the engine reads `~/.local/share/lkasper-hyprland/themes/gruvbox/colors.json` and parses the Base16 palette

#### Scenario: Theme does not exist
- **WHEN** `loadTheme("nonexistent")` is called
- **THEN** the engine logs a warning and does not change the current theme

#### Scenario: List available themes
- **WHEN** `getThemeList()` is called
- **THEN** the engine returns an array of theme names derived from directory names under `~/.local/share/lkasper-hyprland/themes/`

### Requirement: Theme engine generates GTK CSS and applies it
The theme engine SHALL generate a complete GTK CSS string from the loaded palette, inlining all color values. It SHALL use GTK CSS `alpha()` function for transparency (e.g., `alpha(#cdd6f4, 0.12)` instead of SCSS `rgba($text, 0.12)`). The generated CSS SHALL reproduce all styles from the bundle-time `style.scss` with the new color values. It SHALL be applied via `app.reset_css()` followed by `app.apply_css()`.

#### Scenario: Successful CSS generation and application
- **WHEN** a theme is loaded with a valid palette
- **THEN** the engine generates a GTK CSS string with all styles and inlined colors, calls `app.reset_css()` then `app.apply_css()` with the generated CSS, and all AGS windows reflect the new colors

#### Scenario: CSS application is atomic
- **WHEN** `app.reset_css()` and `app.apply_css()` are called in sequence
- **THEN** the transition is near-instant with no visible flash of unstyled content

### Requirement: Theme engine maps Base16 to GTK CSS color values
The generated GTK CSS SHALL map the Base16 palette to the semantic color roles used in the stylesheet. The mapping SHALL be:

| CSS role | Base16 source |
|----------|---------------|
| base (bg) | `#base00` |
| mantle | `#base01` |
| crust | `#base01` |
| surface0 | `#base02` |
| surface1 | `#base03` |
| surface2 | `#base04` |
| overlay0 | `#base04` |
| overlay1 | `#base04` |
| overlay2 | `#base05` |
| subtext0 (fg-dim) | `#base04` |
| subtext1 | `#base05` |
| text (fg) | `#base05` |
| blue (accent) | `#base0D` |
| green | `#base0B` |
| yellow | `#base0A` |
| peach | `#base09` |
| red | `#base08` |
| mauve | `#base0E` |
| teal | `#base0C` |
| lavender | `#base0D` |
| sapphire | `#base0C` |
| sky | `#base0C` |
| pink | `#base0E` |
| flamingo | `#base0F` |
| rosewater | `#base06` |
| maroon | `#base08` |

#### Scenario: Generated CSS uses correct colors
- **WHEN** the theme engine generates GTK CSS from a palette
- **THEN** every color reference in the CSS uses the mapped hex value from the palette, and `alpha()` calls use the hex value directly (e.g., `alpha(#cdd6f4, 0.12)`)

### Requirement: Theme engine propagates colors to Hyprland
The theme engine SHALL write border colors to `~/.config/hypr/theme.conf` in the same format generated by `themes.nix`, then trigger a reload via `hyprctl reload`.

#### Scenario: Hyprland theme file updated
- **WHEN** a theme is applied
- **THEN** `~/.config/hypr/theme.conf` contains `general { col.active_border }` and `group { col.border_active }` using the new palette's base0D (active) and base09 (inactive) with `aa` alpha suffix

#### Scenario: hyprctl not available
- **WHEN** `hyprctl` is not on PATH (e.g., not running under Hyprland)
- **THEN** the engine writes the config file but skips the reload command without error

### Requirement: Theme engine propagates colors to Ghostty
The theme engine SHALL write the terminal color palette to `~/.config/ghostty/themes/lkh-runtime` in Ghostty theme format (matching the format generated by `themes.nix`), then send SIGUSR1 to the Ghostty process to trigger a config reload.

#### Scenario: Ghostty theme file updated and reloaded
- **WHEN** a theme is applied
- **THEN** `~/.config/ghostty/themes/lkh-runtime` contains `background`, `foreground`, `selection-background`, `selection-foreground`, and `palette` entries mapping Base16 slots to terminal color indices, and SIGUSR1 is sent to the Ghostty process

#### Scenario: Ghostty not running
- **WHEN** a theme is applied and no Ghostty process exists
- **THEN** the theme file is written but the SIGUSR1 signal is skipped without error

### Requirement: Ghostty references the runtime theme
`ghostty.nix` SHALL include `theme = "lkh-runtime"` in its settings so Ghostty loads the theme file at `~/.config/ghostty/themes/lkh-runtime`.

#### Scenario: Ghostty uses runtime theme
- **WHEN** Ghostty starts
- **THEN** it loads colors from `~/.config/ghostty/themes/lkh-runtime`

### Requirement: Theme engine propagates colors to btop
The theme engine SHALL write the btop theme file to `~/.config/btop/themes/lkh-runtime.theme` in btop theme format (matching the format generated by `themes.nix`).

#### Scenario: btop theme file updated
- **WHEN** a theme is applied
- **THEN** `~/.config/btop/themes/lkh-runtime.theme` contains `theme[main_fg]`, `theme[hi_fg]`, and other btop color keys using the new palette values

### Requirement: Theme engine propagates colors to starship
The theme engine SHALL write the starship configuration to `~/.config/starship.toml` (matching the format generated by `themes.nix`) with prompt colors derived from the palette.

#### Scenario: starship config updated
- **WHEN** a theme is applied
- **THEN** `~/.config/starship.toml` contains `[directory] style` using base0D, `[git_branch] style` using base05, and `[character] success_symbol` using base0D

### Requirement: Theme engine switches wallpaper via hyprpaper IPC
The theme engine SHALL select the first wallpaper (alphabetically) from `~/.local/share/lkasper-hyprland/themes/<name>/backgrounds/`, then switch wallpaper using hyprpaper IPC: `hyprctl hyprpaper preload <path>` followed by `hyprctl hyprpaper wallpaper ",<path>"`. It SHALL also write the wallpaper path to `~/.config/lkasper-hyprland/current/wallpaper` for use by hyprlock.

#### Scenario: Wallpaper switched on theme change
- **WHEN** a theme with wallpapers in its `backgrounds/` directory is applied
- **THEN** hyprpaper displays the new wallpaper, and `~/.config/lkasper-hyprland/current/wallpaper` contains the absolute path to the image

#### Scenario: Theme has no wallpapers
- **WHEN** a theme is applied that has no files in `backgrounds/`
- **THEN** the existing wallpaper is retained and hyprpaper is not called

### Requirement: Theme switching via AGS IPC
The theme engine SHALL be triggerable via AGS IPC. The `requestHandler` in `app.ts` SHALL accept `theme set <name>` to switch themes, `theme get` to return the current theme name, and `theme list` to return available theme names.

#### Scenario: Switch theme via CLI
- **WHEN** `ags request "theme set gruvbox"` is executed
- **THEN** the theme engine loads gruvbox, applies CSS, propagates to all apps, switches wallpaper, and persists the selection

#### Scenario: Query current theme via CLI
- **WHEN** `ags request "theme get"` is executed
- **THEN** the response contains the current theme name (e.g., `gruvbox`)

#### Scenario: List themes via CLI
- **WHEN** `ags request "theme list"` is executed
- **THEN** the response contains a newline-separated list of available theme names

### Requirement: Mutable config files via home.activation
`themes.nix` SHALL use `home.activation` (not `home.file`) for non-AGS app theme configs that the theme engine needs to overwrite at runtime. The activation script SHALL only write default values if the file does not already exist, preserving runtime theme choices across `home-manager switch`. It SHALL run after the `writeBoundary` phase.

#### Scenario: First boot -- defaults written
- **WHEN** `home-manager switch` runs and `~/.config/hypr/theme.conf` does not exist
- **THEN** the activation script creates the file with tokyo-night default colors

#### Scenario: Subsequent rebuild -- runtime theme preserved
- **WHEN** `home-manager switch` runs and `~/.config/hypr/theme.conf` already exists (written by theme engine)
- **THEN** the activation script does NOT overwrite the file
